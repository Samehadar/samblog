## Проблема

Простой сценарий: нужно переименовать колонку `name` в `full_name` в таблице `users`. В локальной базе это один ALTER TABLE. В продакшне с 10 миллионами записей и 500 запросами в секунду — это потенциальный даунтайм.

Почему? Потому что между моментом, когда миграция применена, и моментом, когда новый код задеплоен, есть окно, в котором старый код пишет в `name`, а колонки уже нет.

## Expand-Contract Pattern

Решение — разделить миграцию на фазы:

**Expand (расширение):**
1. Добавить колонку `full_name`
2. Задеплоить код, который пишет в обе колонки
3. Скопировать существующие данные: `UPDATE users SET full_name = name`

**Contract (сжатие):**
4. Задеплоить код, который читает из `full_name`
5. Задеплоить код, который не пишет в `name`
6. Удалить колонку `name`

Три деплоя вместо одного. Больше работы? Да. Зато ноль даунтайма и возможность откатить на любом шаге.

## Безопасные операции

Не все DDL-операции одинаково опасны. Классификация для PostgreSQL:

**Безопасно (не блокирует):**
- `ADD COLUMN` без NOT NULL и без DEFAULT (в PG 11+, с DEFAULT тоже безопасно)
- `CREATE INDEX CONCURRENTLY`
- `DROP COLUMN` (помечает как невидимую, данные остаются)

**Опасно (берёт блокировку):**
- `ALTER COLUMN SET NOT NULL`
- `ADD COLUMN WITH DEFAULT` (PG < 11)
- `CREATE INDEX` (без CONCURRENTLY)
- `ALTER COLUMN TYPE`

**Очень опасно:**
- `LOCK TABLE`
- Любая операция на большой таблице без `CONCURRENTLY`

## Типичные ловушки

**Большой бэкфилл в транзакции.** UPDATE на 10 миллионов строк в одной транзакции — это lock на всю таблицу. Бейте на батчи по 1000-10000 записей с паузами.

**NOT NULL без значения по умолчанию.** Нельзя просто добавить NOT NULL на существующую колонку. Сначала добавьте DEFAULT, заполните NULL-значения, потом добавьте constraint.

**Foreign key на большую таблицу.** `ADD FOREIGN KEY` валидирует все существующие строки. На таблице с миллионами записей это блокировка на минуты. Используйте `NOT VALID` + `VALIDATE CONSTRAINT` отдельным шагом.

**Удаление колонки, которую читает ORM.** Django, ActiveRecord и другие ORM кэшируют схему. Удалили колонку — ORM падает. Сначала уберите колонку из модели, задеплойте, и только потом удаляйте из базы.

## Инструменты

Не изобретайте велосипед:
- **gh-ost** (GitHub) — онлайн-миграции для MySQL
- **pt-online-schema-change** (Percona) — аналог для MySQL
- **pgroll** — онлайн-миграции для PostgreSQL
- **squawk** — линтер для PostgreSQL-миграций (ловит опасные операции)

Встраивайте проверку миграций в CI. Squawk в пайплайне ловит 90% проблем до того, как они попадут в прод.
