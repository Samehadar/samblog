## Зачем мы делаем code review

Code review существует не для того, чтобы найти баги. Для этого есть тесты. Code review существует для трёх вещей:

1. **Обмен знаниями** — каждый PR увеличивает bus factor проекта
2. **Поддержание стандартов** — консистентность кодовой базы
3. **Раннее выявление архитектурных проблем** — когда менять дёшево

Если ваш code review сводится к «выглядит ок, аппрувлю» — это ритуал, а не процесс.

## Размер PR имеет значение

Исследования показывают: чем больше PR, тем менее внимательно его ревьюят. PR на 500 строк получает детальные комментарии. PR на 2000 строк получает «LGTM».

Оптимальный размер: 200-400 строк изменений. Если фича не влезает — разбивайте на последовательные PR. Да, это требует усилий по декомпозиции. Но эти усилия окупаются качеством ревью.

## Что смотреть на code review

Не всё одинаково важно. Приоритет:

**Критично:**
- Корректность логики
- Безопасность (SQL injection, XSS, утечка данных)
- Обработка ошибок и edge cases

**Важно:**
- Архитектурные решения и их обоснование
- Тестовое покрытие новой логики
- Производительность (N+1 запросы, лишние аллокации)

**Желательно:**
- Именование переменных и методов
- Структура и читаемость
- Соответствие code style (но лучше автоматизировать)

## Антипаттерны

**Nitpicking** — комментарии про пробелы и запятые. Автоматизируйте форматирование (Prettier, Black, gofmt) и забудьте об этом навсегда.

**Гейткипинг** — один человек блокирует PR неделями. Установите SLA: первый ревью — в течение 4 рабочих часов. Финальный аппрув — в течение рабочего дня после исправлений.

**Переписывание** — «я бы сделал по-другому» без объективных причин. Если код работает, читаем и поддерживаем — стиль автора имеет право на существование.

**Rubber stamping** — автоматический аппрув без чтения. Хуже, чем отсутствие ревью, потому что создаёт иллюзию проверки.

## Как писать комментарии

Хороший комментарий на code review содержит три компонента:

1. **Наблюдение**: что вы видите
2. **Причина**: почему это проблема
3. **Предложение**: как можно улучшить

Плохо: «Тут нужен try-catch»

Хорошо: «Вызов `externalApi.fetch()` может бросить `TimeoutException`. Без обработки это уронит весь запрос. Предлагаю обернуть в try-catch и вернуть fallback-значение, как мы делаем в `OrderService`»

## Автоматизация

Всё, что можно проверить автоматически, не должно проверяться человеком:

- **Линтеры и форматтеры** — ESLint, Prettier, Pylint, Clippy
- **Статический анализ** — SonarQube, CodeClimate
- **Тесты** — unit, integration, e2e в CI
- **Security scanning** — Snyk, Dependabot
- **Проверка размера PR** — бот, который предупреждает о PR > 500 строк

Чем больше автоматизировано, тем больше времени у ревьюера на действительно важные вещи: логику, архитектуру и обмен знаниями.
