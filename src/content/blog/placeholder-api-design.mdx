## Почему API-дизайн важнее реализации

API — это контракт. Реализацию можно переписать за спринт. Контракт, от которого зависят десять клиентов, переписать нельзя.

Я видел проекты, где команда потратила три месяца на идеальную архитектуру сервиса, но API проектировалось в последний день. Результат предсказуем: через полгода половина эндпоинтов обросла костылями, а вторая половина была deprecated.

## Принцип минимального удивления

API должен вести себя так, как ожидает разработчик, который видит его впервые. Это значит:

- **Консистентные имена**: если `GET /users/{id}` возвращает пользователя, то `GET /orders/{id}` должен возвращать заказ, а не список заказов
- **Предсказуемые коды ответов**: 404 — не найдено, 400 — кривой запрос, 409 — конфликт. Не 200 с телом `{"error": "not found"}`
- **Единообразная пагинация**: один подход для всех списковых эндпоинтов

## Версионирование: когда и как

Версионирование API — это не вопрос «нужно ли», а вопрос «когда начнёте». Три основных подхода:

| Подход | Пример | Плюсы | Минусы |
|---|---|---|---|
| URL path | `/v1/users` | Очевидно, легко роутить | Дублирование роутов |
| Header | `Accept: application/vnd.api+json;version=2` | Чистые URL | Сложнее тестировать |
| Query param | `/users?version=2` | Простота | Кэширование сложнее |

Для большинства проектов URL path — оптимальный выбор. Он прозрачен, его видно в логах, его легко объяснить джуниору.

## Обратная совместимость

Золотое правило: **добавлять можно, удалять и менять нельзя**.

Безопасные изменения:
- Добавить новое поле в ответ
- Добавить новый эндпоинт
- Добавить опциональный параметр запроса

Ломающие изменения:
- Удалить поле из ответа
- Переименовать поле
- Изменить тип поля
- Сделать опциональный параметр обязательным

Если нужно ломающее изменение — это повод для новой версии.

## Документация как код

Swagger/OpenAPI — не опция, а необходимость. Но есть нюанс: спецификация должна быть source of truth, а не артефакт.

Два подхода:
1. **Code-first**: пишете код, генерируете спеку. Быстро, но спека отстаёт
2. **Spec-first**: пишете спеку, генерируете код/стабы. Медленнее, но контракт первичен

Для публичных API — только spec-first. Для внутренних — допустим code-first с автотестами на контракт.

## Обработка ошибок

Хороший API сообщает не только что пошло не так, но и почему, и что с этим делать:

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Request validation failed",
    "details": [
      {
        "field": "email",
        "reason": "Must be a valid email address",
        "value": "not-an-email"
      }
    ]
  }
}
```

Плохой API возвращает `{"error": "Bad request"}` и оставляет клиента гадать.

## Rate limiting и graceful degradation

Любой публичный API должен иметь rate limiting с первого дня. Не потому что вы ожидаете нагрузку, а потому что один баг в клиентском коде может положить ваш сервис.

Стандартные заголовки:
- `X-RateLimit-Limit` — лимит запросов
- `X-RateLimit-Remaining` — оставшиеся запросы
- `X-RateLimit-Reset` — когда сбросится счётчик
- `Retry-After` — при 429 статусе

Клиент, получивший 429, должен точно знать, когда можно повторить запрос.
